# clisbot v0.1.44

Released: 2026-04-25

`v0.1.44` focuses on the shared-team security model: clearer group and DM route naming, safer allowlist and blocklist behavior, automatic config migration from `0.1.43`, and better CLI help for AI-assisted operators.

## In One Line

`clisbot` now has a cleaner security and routing contract for Slack and Telegram teams, while existing `0.1.43` installs upgrade automatically with a backup.

## Highlights

### 1. Shared group access is easier to reason about

- CLI route ids now standardize on `group:<id>`, `group:*`, `dm:<id>`, `dm:*`, and `topic:<chatId>:<topicId>`.
- Stored bot config uses `directMessages` and `groups` with raw ids plus `*`, so config files stay provider-scoped and less noisy.
- Slack `channel:<id>`, `groups:*`, and older wildcard forms remain accepted as compatibility input.

### 2. Admission policy and sender policy are separated

- `groupPolicy` and Slack `channelPolicy` decide whether a multi-user surface is admitted.
- `groups["*"].policy` and exact `groups["<id>"].policy` decide who can speak inside admitted groups.
- Adding a group route under the default `allowlist` admission policy now makes that group usable immediately with sender policy `open`.

### 3. Allow and block users now work for real team use cases

- `allowUsers` and `blockUsers` are enforced before the runner sees a shared group prompt.
- `group:*` can allow or block a user across every admitted group under a bot.
- `disabled` means fully silent on that surface, including owner/admin and pairing guidance.
- Owner/admin do not bypass group admission, but they can bypass sender allowlists after a group is admitted; `blockUsers` still wins.

### 4. Existing `0.1.43` configs auto-upgrade safely

- On first read of a released `0.1.43` config, clisbot backs up the original file under `backups/`.
- It then validates the `0.1.44` shape before writing the upgraded config.
- Upgrade logs show each stage: backup, preparation, dry-run validation, apply, and success.
- If the config is already on the current schema, the upgrade path is skipped without backup or upgrade logs.

### 5. CLI help is better for humans and AI agents

- `clisbot routes help` now explains canonical route ids, legacy-compatible inputs, policy values, and exact examples for common group and DM mutations.
- Route policy validation rejects invalid values before writing config.
- Docs and user guides now front-load the security mental model so AI-assisted setup can discover the right commands from local help and docs.

### 6. Gemini compatibility status is corrected

- README now marks Gemini as fully compatible for routed Slack and Telegram workflows.
- The previous "usable with caveats" label was too conservative for the current runner support.

## Upgrade Notes

- Config schema is now `0.1.44`.
- Existing `0.1.43` configs are upgraded automatically on first read.
- A backup is written before the upgraded config is applied, for example:

```text
~/.clisbot/backups/clisbot.json.0.1.43.<timestamp>
```

- After the config is rewritten to `0.1.44`, downgrading to `0.1.43` should use that backup.
- Dedicated downgrade and restore CLI UX is tracked separately in the backlog.

## Validation

Validated with:

- `bun run check`

And supported by targeted regressions for:

- `0.1.43` to `0.1.44` config migration
- auto-backup and stage logging during upgrade
- no-op behavior when schema is already current
- Slack and Telegram shared-route audience enforcement
- canonical and legacy route ids
- wildcard `group:*` allow/block user mutations
- route policy validation
- CLI help content for route ids and policy examples
